# Private To Public Publishing Flow

## Why It Matters

A public memory layer is most useful when it is consistent, reviewable, and clearly downstream from private canonical memory. A simple publishing flow helps preserve that relationship.

## Public-Safe Summary

The safest pattern is to author operational truth privately, then create public notes as rewritten derivatives. This prevents the publishable layer from becoming a second competing source of truth and makes redaction a deliberate step instead of an afterthought.

## Flow

1. Write or update the canonical note in private memory.
2. Identify whether the note contains a reusable public lesson, pattern, prompt, workflow, or decision.
3. Draft a public version that explains the idea without depending on private operational context.
4. Rewrite examples, names, paths, and references so the note stands on its own as a public artifact.
5. Run a redaction review before treating the draft as publishable.
6. Link the public note to related public architecture, workflow, prompt, or decision notes.
7. Publish or export from the public layer only after review.

## Pattern

- Keep private memory canonical.
- Treat public memory as a derived layer.
- Rewrite for publication instead of copying private notes directly.
- Review for both safety and usefulness.
- Publish only the minimum detail needed to teach the lesson.

## Boundaries

- This flow is not meant for raw exports, logs, screenshots, or transcripts.
- Public publishing should not force private notes to be written as if they were already public.
